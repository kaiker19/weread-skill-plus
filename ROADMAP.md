## 官方接口全景（weread-skills v1.0.3）

| 分类 | 接口 | 说明 | 潜在挖掘点 |
|------|------|------|-----------|
| 搜索 | `/store/search` | 按书名/作者/全文/书单等多 scope 搜索 | — |
| 书籍 | `/book/info` | 书籍详情（书名、作者、评分、分类、简介） | category 字段可做兴趣图谱分类 |
| 书籍 | `/book/chapterinfo` | 章节目录（层级、字数、付费状态） | 配合进度可知道"卡在哪章" |
| 书籍 | `/book/getprogress` | 阅读进度（0-100%）、累计时长、读完时间 | `finishTime` 是触发读后总结的精准信号 |
| 书架 | `/shelf/sync` | 全书架（电子书 + 有声书 + 公众号合集） | 在读/未读/完读分布；书籍分类聚合 |
| 阅读统计 | `/readdata/detail` | 周/月/年/总 阅读时长、天数、偏好分类、最多读的书 | `preferCategory` 反映兴趣结构；日均时长变化趋势 |
| 笔记 | `/user/notebooks` | 有笔记的书列表（按 lastSort 游标分页） | 笔记数量 = 参与深度指标 |
| 笔记 | `/book/bookmarklist` | 单书全部个人划线 | 核心知识内容来源 |
| 笔记 | `/review/list/mine` | 个人想法/批注（自己写的文字，非划线） | 优先级高于划线，是原创思考的载体 |
| 笔记 | `/book/underlines` | 章节划线热度统计（多少人划了这段） | 识别本书的"共识核心章节" |
| 笔记 | `/book/bestbookmarks` | 大众热门划线内容（含划线文本） | 与个人划线对比，识别盲点；`chapterUid=0` 取全书 |
| 点评 | `/review/list` | 某本书的公开点评列表 | 读完时对比他人点评，辅助形成自己的观点 |
| 发现 | `/book/recommend` | 基于阅读记录的个性化推荐（"为你推荐"） | 结合当前兴趣主题筛选推荐 |
| 发现 | `/book/similar` | 基于某本书的相似推荐 | 读完一本书后的延伸阅读建议 |

## 用户类型

| 类型 | 特征 | 主要触点 |
|------|------|---------|
| **A 类：有 Agent**（OpenClaw/Hermes 用户） | 接受主动推送，Agent 嵌入日常工作流 | Agent 推送为主，Web 为可视化补充 |
| **B 类：无 Agent**（纯 Web 用户） | 主动打开 Web，无推送 | Web 是完整产品，需承担 Agent 的总结能力 |

## Aha Moment → 需求对应

每个 aha moment 是某个功能的情绪回报，也反向验证了这个功能值得做。

| Aha Moment | 对应需求 |
|-----------|---------|
| **读后洞见**：总结里有一句话让你觉得"说得对，但我没这样想过" | Agent 读后总结 |
| **知识轨迹**：图谱里看到自己三年来一直围绕几个相近主题打转 | Web 知识图谱可视化 |
| **跨书回声**：今天划的一段，呼应了 8 个月前另一本书里的批注 | Agent 每日总结（语义匹配历史） |
| **盲点发现**：这本书被最多人划的段落，你跳过了 | Agent 读后总结（对比 bestbookmarks） |
| **知识被用上**：做某件事时 Agent 自动浮现相关历史划线，且真的有用 | Agent 阅读记忆注入（长期方向） |

## 需求

### Agent 端（A 类用户）

核心：**日常推送 + 历史关联**，不是功能堆砌，是每天都想看的内容。

**每日总结**（定时触发）
- 当天新增划线 + 批注 → LLM 提炼"今天在读什么、在想什么"
- 检测语义重叠，附上跨书呼应（有则说，无则不提）
- 没有新内容时：静默，不推空消息

**读后总结**（事件触发：`finishTime` 出现）
- 基于整本书的所有划线 + 批注，生成"我从这本书里得到了什么"
- 与大众热门划线对比：你的独特视角 vs 你可能错过的共识内容
- 可附延伸推荐（`/book/similar`）
- 比每日总结更完整、更有仪式感，是一次"收尾"

> **两者的核心区别**：每日总结是"此刻在想什么"（横截面），读后总结是"这本书给了我什么"（纵深回顾）

### Web 端（A 类 + B 类用户）

B 类用户没有 Agent，Web 需要承担所有能力。核心三层，由浅到深：

**第一层：阅读全貌可视化**
- 书架状态：在读 / 完读 / 搁置，时间轴上的完读记录
- 每本书参与深度：划线数、批注数、读完与否
- 阅读节奏曲线：周/月时长趋势，看到自己的阅读习惯

**第二层：知识结构可视化**
- 以 LLM 提炼的主题/概念为节点，书为聚合单元
- 同主题跨书划线聚合展示
- 时间维度：兴趣随时间的迁移轨迹

**第三层：沉淀与输出**
- 基于知识库做主题综述（辅助写作/思考）
- 交互式内容生成：选定主题 → 生成有观点的素材

## 底座：共享语义知识库

Agent 增量写入，Web 查询展示，两端共用。

**核心数据**（优先级排序）：
1. `review/list/mine` — 个人批注（原创思考，权重最高）
2. `book/bookmarklist` — 个人划线（筛选后的他人文字）
3. `book/bestbookmarks` — 大众热门划线（对比用，非主库）
4. `book/getprogress` — 完读信号（触发总结的事件源）

**本地实现选型（Agent 端 skill）**

向量检索的前提是 embedding，embedding 有两种来源：
- OpenAI API：需要用户配置 key，有网络调用
- 本地模型（sentence-transformers）：需下载 100MB~1GB 模型

两者都要用户额外配置，对本地 skill 来说是负担。**Agent 端推荐绕过向量检索**：

| 方案 | 依赖 | 适用 |
|------|------|------|
| **SQLite + LLM 提取概念标签** | 零（Python 内置 sqlite3） | Agent 端首选；LLM 本来就可用，提取的标签直接存 SQLite，跨书匹配用标签 + LLM 判断 |
| SQLite + sqlite-vec | 需安装 sqlite-vec 扩展 + embedding API | 需要精确语义检索时升级 |
| ChromaDB | 依赖重（内部也用 SQLite），仍需 embedding | 不推荐用于本地 skill |

**Web 端**：服务端统一配置 embedding API，用户无感知，可直接用向量检索（ChromaDB / Qdrant 均可）。
